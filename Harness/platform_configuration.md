# Platform Configuration And Build Surface

## Goal

Document the package manifest, Xcode target, scheme, template, resource, and validation surfaces agents must touch deliberately.

## When To Load

- Editing `Package.swift`, Xcode project settings, deployment targets, schemes, app targets, templates, or resources.
- Diagnosing package or app build failures caused by platform availability.
- Adding native platform support.

## Current Build Surface

Run `xcodebuild -list -project FuturamaInvadersNew.xcodeproj` to discover the current project state.

Known app schemes:

| Scheme | Target | Use |
| --- | --- | --- |
| `SPM-Modular-Programming` | `FuturamaInvadersNew` | iOS-family app build and run surface. |
| `FuturamaInvadersNew-macOS` | `FuturamaInvadersNew-macOS` | Native macOS app build and run surface. |
| `FuturamaInvadersNew-visionOS` | `FuturamaInvadersNew-visionOS` | Native visionOS app build and run surface. |

| Scheme | Package |
| --- | --- |

| `Detail` | `FeaturePackages/Detail` |

## Platform Rules

- Package manifests declare iOS 26, macOS 26, tvOS 26, watchOS 26, and visionOS 26.
- The iOS app target uses `SDKROOT = iphoneos` and must not include `macosx` or `xros` in `SUPPORTED_PLATFORMS`.
- The native macOS target uses `SDKROOT = macosx` and `SUPPORTED_PLATFORMS = macosx`.
- The native visionOS target uses `SDKROOT = xros`, `SUPPORTED_PLATFORMS = "xros xrsimulator"`, `TARGETED_DEVICE_FAMILY = 7`, and `XROS_DEPLOYMENT_TARGET = 26.0`.
- Keep the iOS launch storyboard out of macOS and visionOS resource phases.
- Do not use Catalyst or Mac Designed for iPad settings when the task asks for native macOS.

## Environment And Traits

Package manifests use `IS_RELEASE` to decide the default `DEBUG` trait:

```swift
let isDebug = ProcessInfo.processInfo.environment["IS_RELEASE"] != "true"
```

- Debug/local package work should usually run with `env -u IS_RELEASE` so mocks are available.
- Release/profile schemes should set `IS_RELEASE=true` when the intent is to exclude mock targets.
- Do not pass secrets through environment variables in project files or schemes.

## Resources

- App resources live in `App/Assets.xcassets` and the iOS launch storyboard lives at `App/LaunchScreen.storyboard`.
- Package resources live under `Sources/Main/Resources` and must be declared with `resources: [.process("Resources")]` in the owning target.
- Mock response fixtures live under `Sources/Mocks/Responses` and are processed by mock targets.
- Package dummy color sets exist to force package bundle generation; do not treat them as design tokens.

## Xcode Templates

- `Xcode Templates/Module.xctemplate` should match live package manifests: all Apple OS platforms, `IS_RELEASE` trait gating, `Sources/Mocks`, `Sources/Concurrent`, `Sources/Main`, and resource processing.
- `Xcode Templates/View & ViewModel.xctemplate` should match live Observation ViewModels and router-binding patterns.
- Always compare generated template output against the nearest live package before committing it.

## Validation Commands

Use the narrowest command that covers the touched surface:

```sh
xcodebuild -list -project FuturamaInvadersNew.xcodeproj
env -u IS_RELEASE xcodebuild build -project FuturamaInvadersNew.xcodeproj -scheme SPM-Modular-Programming -destination 'generic/platform=iOS Simulator'
env -u IS_RELEASE xcodebuild build -project FuturamaInvadersNew.xcodeproj -scheme FuturamaInvadersNew-macOS -destination 'platform=macOS'
env -u IS_RELEASE xcodebuild build -project FuturamaInvadersNew.xcodeproj -scheme FuturamaInvadersNew-visionOS -destination 'generic/platform=visionOS'
```

SwiftPM smoke builds from package directories:

```sh
cd SharedPackages/Tools && env -u IS_RELEASE swift build
cd FeaturePackages/Detail && env -u IS_RELEASE swift build
cd FeaturePackages/Entry && env -u IS_RELEASE swift build
```

visionOS, tvOS, and watchOS builds require matching Xcode platform components. If Xcode reports a platform is not installed, document that environment blocker rather than treating it as a source failure.

## Accuracy Contracts

### Do

- Keep app targets separated by native platform.
- Update shared schemes when adding app targets.
- Update templates when changing package structure or ViewModel conventions.
- Validate project parsing after editing `project.pbxproj`.

### Do Not

- Do not put iOS storyboard resources into macOS or visionOS targets.
- Do not edit Xcode user data unless the task explicitly requires it.
- Do not rely on app icon settings for platforms whose asset catalog entries are absent.
